# Development Workflow

## Tooling

Use:

- `uv` for Python dependency management
- `npm` for Tailwind CSS build dependencies
- `just` for local commands
- Ruff for Python formatting and linting
- pytest for fixture-based compiler/content behavior tests
- Pydantic for strict content schema validation
- GitHub Actions for CI and deployment

Node/npm is included only to run Tailwind CSS. The website itself is static and generated by Python.

Use npm rather than pnpm for v1. Tailwind is the only Node-based tool, npm is already available in the expected environment, and `npm ci` with `package-lock.json` is reproducible enough for this small dependency surface. pnpm can be reconsidered later if the project grows into a larger JavaScript or monorepo setup.

## Why npm Is Used

Tailwind scans source templates/content for utility classes and generates an optimized CSS file. It should not scan `dist/`, because that couples CSS generation to previous compiler output and can hide stale/missing classes.

```text
HTML/templates/Markdown using Tailwind classes
        ↓
Tailwind CLI via npm
        ↓
dist/assets/styles.css
```

npm is not used to run the website. It is only a small build dependency for CSS generation.

Recommended npm dependencies:

- `tailwindcss`
- `@tailwindcss/typography`

Tailwind content sources should include:

- `templates/**/*.html`
- `content/**/*.md`

Avoid dynamic Tailwind class construction in Python. If dynamic class strings become unavoidable, make them explicit and covered by Tailwind content configuration or safelisting.

## Local Commands

Use a `justfile` instead of a Makefile. `just` is the canonical local workflow; underlying `uv` and `npm` commands may appear in CI and troubleshooting docs, but day-to-day documentation should not present competing command interfaces.

Recommended commands:

```just
install:
    uv sync
    npm install

build:
    uv run python -m portfolio_site.build
    npm run css

serve:
    python -m http.server 8000 -d dist

dev: build serve

check:
    uv run ruff format --check .
    uv run ruff check .
    uv run pytest
    uv run python -m portfolio_site.build
    npm run css
    test -f dist/index.html
    test -f dist/blog/index.html
    test -f dist/assets/styles.css

format:
    uv run ruff format .
    uv run ruff check --fix .
```

Expected usage:

```bash
just install
just build
just serve
```

or:

```bash
just dev
```

Preview locally at:

```text
http://localhost:8000
```

For v1, `just dev` can build once and serve. File watching can be added later.

## Static Type Checking

Do not include mypy or pyright in v1. Pydantic, pytest, and Ruff cover the main early risks, while a type checker would add configuration and annotation overhead before the compiler shape is stable. Reconsider static type checking after v1 if the Python code grows.

## Content Validation Approach

Use Pydantic models for structured YAML content and Markdown frontmatter validation. The content model has enough strict schemas, controlled fields, URL/date patterns, typed links, and unknown-field rejection that centralized model validation is more maintainable than scattered manual checks.

## Compiler Module Boundaries

Keep compiler responsibilities separated:

- `content.py`: load and validate domain/source content
- `routes.py`: own public URL to output path inventory
- template/rendering layer: render HTML from validated content; homepage `index.html` composes section partials rather than inlining every section
- `build.py`: orchestrate validation, rendering, clean output, asset copying, and CSS handoff

`build.py` should not become a god script. Route inventory should be the source of truth for Markdown internal link validation and generated output paths.

## Build Pipeline

Use a compiler-style build, not loose independent generation scripts.

```text
Markdown + YAML + assets
        ↓
load source content
        ↓
validate schemas, slugs, URLs, links, and declared assets
        ↓
prepare rendered pages
        ↓
remove existing dist/ after validation succeeds
        ↓
write HTML files and copy full assets/ tree
        ↓
Tailwind CLI writes CSS
        ↓
GitHub Pages deployment
```

Recommended build commands:

```bash
uv run python -m portfolio_site.build
npm run css
```

The site compiler should avoid stale output by performing full clean builds. It should validate before deleting or writing `dist/`, then write fresh output directly to `dist/`. V1 does not use incremental rendering or cached Markdown/rendered output because the site is small and cache invalidation would add reliability risk. V1 also does not need a temporary output directory plus atomic swap because `dist/` is disposable generated output and simpler build behavior is preferred.

## Build Validation

The build should fail loudly when required content is invalid. Structured content and frontmatter use strict schemas; unknown fields fail.

### Cross-cutting validation

Fail if:

- content contains unknown YAML/frontmatter fields
- content tags are not lowercase kebab-case
- local asset references do not start with `/assets/`
- local asset references contain `..`
- declared local asset references do not resolve inside `assets/`
- Markdown images use external URLs or unresolved local assets
- Markdown internal links starting with `/` do not resolve to a generated page or bundled asset

External links must be `http://` or `https://`, except provider-specific profile links, which are checked more strictly. V1 does not check external link reachability.

### Blog validation

Every blog file must have:

- `title`
- `status`

Fail if:

- `status` is not `published` or `draft`
- the filename stem is not lowercase kebab-case
- blog files are nested below `content/blog/`
- duplicate slugs exist

For `status: published`, also require:

- `published_date`
- `summary`

For `status: published`, validate:

- `published_date` is a valid date
- optional `updated_date` is a valid date
- `updated_date` is not earlier than `published_date`
- optional `links` contains only `github` and/or `demo`

Draft behavior:

- `status: draft` posts are skipped from output.
- drafts may omit `summary`, `published_date`, and `updated_date`.
- drafts must still have valid slugs.

### Profile validation

Fail if missing:

- `name`
- `avatar`
- `hero_lines`
- `links.github`
- `links.linkedin`

Also validate:

- `hero_lines` has at least one line
- avatar is a valid local asset reference
- GitHub link uses the expected GitHub profile URL shape
- LinkedIn link uses the expected LinkedIn profile URL shape
- no extra profile links are present in v1

### Technology marquee validation

Fail if:

- a row is malformed
- a row is missing its source label
- an item is missing `name`
- an `icon` path is provided but is not a valid local SVG asset reference

Icons are optional. Marquee row labels are source organization only and should not appear as visible headings in v1.

### Papers validation

Required paper fields:

- `title`
- `year`
- `summary`
- at least one paper artifact link: `links.pdf` or `links.external`

Validate:

- `links.pdf` is a valid local asset reference
- `links.external` is an `http://` or `https://` URL
- optional `links.github` is an `http://` or `https://` URL
- code links do not satisfy the paper artifact requirement

### Books validation

Required book fields:

- `title`
- `author`
- `status`: `read` or `reading`

Fail if:

- status is `want_to_read` or any other value
- fields like `summary`, `reflection`, or `year_read` are present

### Certificates validation

Required certificate fields:

- `title`
- `issuer`
- `issued_date`: `YYYY`, `YYYY-MM`, or `YYYY-MM-DD`
- `links.pdf`

Validate:

- optional `tags` are lowercase kebab-case content tags
- `links.pdf` starts with `/assets/`, contains no `..`, resolves inside `assets/`, exists, and has a `.pdf` suffix
- unknown fields fail, including unsupported `summary` or credential verification links

For sorting, treat partial dates as the latest possible date at that precision.

### Events validation

Required event fields:

- `name`
- `role`: exactly one of `attendee`, `speaker`, `organizer`, or `volunteer`
- `date`: `YYYY`, `YYYY-MM`, or `YYYY-MM-DD`

Validate:

- optional `location` is free display text
- optional `tags` are lowercase kebab-case content tags
- optional `links.homepage` is an `http://` or `https://` event homepage URL
- optional `links.pdf` starts with `/assets/`, contains no `..`, resolves inside `assets/`, exists, and has a `.pdf` suffix
- unknown fields fail

For multi-day events, `date` is the start date. For sorting, treat partial dates as the latest possible date at that precision.

## Testing Approach

Use small fixture site directories for tests instead of mocking filesystem/content loaders. Fixtures should exercise real YAML, Markdown, assets, route generation, and output paths so integration bugs appear in the same places they would during a real build.

Prefer tests of external compiler behavior over private helper calls.

## Output Rules

Generate pretty directory-style HTML paths:

```text
/              → dist/index.html
/blog/         → dist/blog/index.html
/blog/<slug>/  → dist/blog/<slug>/index.html
```

Generate `/blog/` even when there are zero published posts. Hide empty homepage content sections rather than showing placeholders.

Copy the full `assets/` directory to `dist/assets/`, including unreferenced files.

## CI/CD for v1

Include two GitHub Actions workflows. Pin runtime versions to Python `3.14` and Node `v26.0.0` to reduce CI drift.

### Check workflow

Runs on pushes and pull requests.

Checks:

```text
uv sync --locked
npm ci
uv run ruff format --check .
uv run ruff check .
uv run pytest
uv run python -m portfolio_site.build
npm run css
test -f dist/index.html
test -f dist/blog/index.html
test -f dist/assets/styles.css
```

Purpose:

- Verify code formatting
- Verify linting
- Verify behavior tests
- Verify content validation
- Verify site can build

### Deploy workflow

Runs on pushes to `main` and must depend on the check workflow passing. Deployment is not best-effort: if strict validation, link checks, linting, formatting, or the full site build fails, nothing is published.

Steps:

```text
uv sync --locked
npm ci
uv run python -m portfolio_site.build
npm run css
upload dist/ to GitHub Pages
```

`dist/` should be generated in CI and published as the GitHub Pages artifact.

## Files to Commit

Commit:

- source Python files
- templates
- content
- assets
- `.python-version`
- `.node-version`
- `pyproject.toml`
- `uv.lock`
- `package.json`
- `package-lock.json`
- `tailwind.config.js`
- `justfile`
- GitHub Actions workflows

Lockfiles must be committed and CI should fail if `uv.lock` or `package-lock.json` is missing or stale. Use `uv sync --locked` and `npm ci` in CI.

Do not commit:

- `dist/`
- Python caches
- `node_modules/`

## Post-v1 Tooling Backlog

Defer until after the first working site:

- Dependabot
- pre-commit hooks
- external link checker
- accessibility checker
- Lighthouse/performance checks
- mypy or pyright static type checking if the Python codebase grows
- custom-domain/CNAME support
